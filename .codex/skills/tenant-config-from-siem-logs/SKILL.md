# Tenant Config From SIEM Logs

Use this skill to write or revise tenant-layer configuration under `tenants/<tenant>/` from real logs or event samples observed on the SIEM.

## Start With Repo Truth

Read only the files needed for the requested tenant change.

- Read `README.en.md` for the high-level repository model when the task spans multiple layers.
- Read `docs/architecture/tenants-relationship.md` before changing tenant structure.
- Read `docs/architecture/mappings-relationship.md` when the task touches canonical fields or field bindings.
- Read the relevant existing rules under `rules/` to understand which `logsource` patterns and dataset semantics already drive deployable content.
- Read nearby files under the target tenant first so new files match the repository's live naming and YAML style.
- Use `tenants/lab/` as the style baseline when the target tenant or device does not yet have equivalent files.

## Scope Of This Skill

Use this skill when the input is real SIEM data and the goal is to make the tenant layer usable for rendering or deployment.

This skill may create or update:

- `tenants/<tenant>/tenant.yaml`
- `tenants/<tenant>/devices/*.yaml`
- `tenants/<tenant>/logsources/*.yaml`
- `tenants/<tenant>/bindings/ingest/*.yaml`
- `tenants/<tenant>/bindings/fields/*.yml`
- `tenants/<tenant>/filters/**`

Prefer editing existing files over creating parallel duplicates when a matching device or dataset already exists.

## Required Inputs

Gather the minimum inputs needed to build tenant config from SIEM logs.

- `tenant_id`
- `siem_id`
- the real SIEM log or event sample
- vendor, product, and device context if known
- the target device or logsource if it already exists
- the relevant dataset or ingest split if it already exists
- the rule or rule pack intended for this tenant, if deployment scope is part of the task

Infer from existing tenant files when safe. Ask only for missing blockers that cannot be derived from repo state or the provided events.

## Large Input Handling

Keep context bounded when the input files or event samples are large.

- Start with a self-imposed limit of about 200 lines or 16 KB per large file or pasted event sample.
- Prefer reading one representative slice first, such as the header plus the first meaningful event block, instead of loading the whole file.
- Expand to the next slice only when the first slice is not enough to determine `device_id`, `dataset_id`, `index`, `sourcetype`, or the fields needed for mapping.
- When the input is a long event export, prioritize the parts that expose parser metadata, field names, and repeated event structure.
- When the input is a large repo file, use targeted reads around the relevant section instead of opening the entire file.
- State briefly when a large input was sampled instead of read in full.

## Repository Workflow

Describe and follow the tenant workflow the same way the repo documents it.

1. Read `tenant.yaml` to confirm `tenant_id`, `siem_id`, timezone, and general tenant context.
2. Read `devices/*.yaml` to identify the tenant devices already modeled.
3. Read `logsources/*.yaml` to identify the logical datasets declared for each `device_id`.
4. Read `bindings/ingest/*.yaml` to see how each `dataset_id` is currently routed to real SIEM ingest targets such as `index` and `sourcetype`.
5. Read `bindings/fields/*.yml` to see how canonical fields are mapped to actual SIEM fields.
6. Read generated deployment outputs only when needed for context, but do not create or edit `deployments/rule-deployments.yaml` in this skill.
7. Read `filters/` and `overrides/` only when the task includes tenant-specific tuning or render-time narrowing.

Treat this as the authoritative direction of the current repo flow for tenant onboarding:

`tenant.yaml -> devices -> logsources -> bindings/ingest -> bindings/fields -> generated deployments/artifacts`

## End-To-End Workflow

Follow this order.

1. Inspect the target tenant directory and determine which files already exist for the device or product.
2. Classify the incoming SIEM logs into the right `device_id` and one or more `dataset_id` values.
3. Update or create `devices/*.yaml` so the tenant has the correct device identity and product metadata.
4. Inspect the relevant existing rules and reuse their `logsource` expectations together with the real log shape to choose `dataset_id` names that meaningfully indicate which rules can apply to that logsource.
5. Update or create `logsources/*.yaml` so the logical datasets, category, and log type reflect both the observed events and the dataset semantics required by the existing rules.
6. Update or create `bindings/ingest/*.yaml` so each dataset maps to the real SIEM ingest target, and derive `index` and `sourcetype` from the actual SIEM logs or explicit SIEM event metadata instead of guessing from vendor or product names.
7. If the task requires canonical field bindings, use `$tenant-field-binding-writer` to write or update `bindings/fields/*.yml` instead of inventing field-binding logic here.
8. If the relevant canonical mapping file under `mappings/` is missing or incomplete, use `$detection-mapping-ocsf` to update that mapping layer before finishing the tenant field binding.
9. Update tenant `filters/` only when the user explicitly needs tenant-specific rule selection or narrowing.
10. Re-read every touched file and verify cross-file consistency for `tenant_id`, `siem_id`, `device_id`, `dataset_id`, and ingest routing.

## File Decision Rules

- Keep tenant-wide metadata in `tenant.yaml`.
- Keep product or device identity in `devices/*.yaml`.
- Keep logical dataset definitions in `logsources/*.yaml`, and choose `dataset_id` names that remain useful as keys for matching the existing rule inventory.
- Keep physical SIEM ingest routing in `bindings/ingest/*.yaml`, and source `index` plus `sourcetype` from the real SIEM logs.
- Keep canonical-to-SIEM field resolution in `bindings/fields/*.yml`.
- Keep tenant-specific selection logic in `filters/`, not in base rules.
- Treat `deployments/rule-deployments.yaml` as generated output managed by repository code, not a hand-edited tenant source file in this workflow.

## How To Write Each File

Write files in the same shape and naming style already used under `tenants/lab/`.

### `tenant.yaml`

- Keep one root tenant file at `tenants/<tenant>/tenant.yaml`.
- Include at least `schema_version`, `tenant_id`, `name`, `environment`, `timezone`, and `siem_id`.
- Add `default_index` only when the tenant really has a meaningful default.
- Keep extra operator context under `metadata`.

### `devices/device_<device>.yaml`

- Use one device file per `device_id`.
- Follow the live naming pattern `device_<product_or_platform>.yaml`.
- Include at least `schema_version`, `tenant_id`, `device_type`, `device_id`, `vendor`, and `product`.
- Add `category`, `role`, `functions`, `location`, `network_zone`, `description`, and `criticality` when known or when nearby files already carry those fields.
- Keep `device_id` stable and hyphenated because it is the anchor key reused by `logsources` and `bindings`.

### `logsources/logsource_<device>.yaml`

- Use one logsource file per `device_id`.
- Follow the live naming pattern `logsource_<product_or_platform>.yaml`.
- In the current repo state, prefer the `datasets:` form unless the task clearly belongs to the schema's alternate `service:` pattern.
- Each dataset object should include `dataset_id`, `description`, `category`, `log_type`, and `enabled`.
- Choose `dataset_id` from both the real log split and the rule inventory, because this key is how the tenant data lines up with applicable detection content.
- Reuse existing dataset naming patterns when the meaning matches, such as vendor-specific alert datasets or compact dataset keys like `traffic`, `api`, `app`, or `system`.

### `bindings/ingest/binding_<device>.yaml`

- Use one ingest binding file per `device_id`.
- Follow the live naming pattern `binding_<product_or_platform>.yaml`.
- Include `schema_version`, `tenant_id`, `device_id`, and `siem_id`.
- In the current repo examples, prefer the `datasets:` array form with `dataset_id`, `index`, and `sourcetype`.
- Ensure every `dataset_id` here exists in the corresponding logsource file.
- Fill `index` and `sourcetype` from the real SIEM logs or event metadata, not from product assumptions.

### `bindings/fields/*.yml`

- Do not hand-roll this logic in this skill.
- Use `$tenant-field-binding-writer` for canonical-to-SIEM field binding.
- Keep the tenant field binding scoped by `device_id`, with dataset-specific overrides only when field names truly differ by dataset.

### `filters/` and `overrides/`

- Only create or update these files when the user explicitly needs tenant-specific narrowing or tuning.
- Keep them as delta files, not replacements for the base rule or tenant identity files.
- Preserve the existing folder style already present under `tenants/lab/overrides/execution/` and `tenants/lab/overrides/filter/`.

## Mapping And Binding Coordination

- Treat `mappings/` as the canonical vocabulary layer.
- Treat `tenants/.../bindings/fields/` as the tenant's physical SIEM field layer.
- Do not define new canonical semantics directly inside tenant files.
- When field binding work is required, explicitly use `$tenant-field-binding-writer`.
- When the field binding skill discovers missing canonical mappings, explicitly use `$detection-mapping-ocsf` before returning to the tenant task.
- Keep the final tenant config minimal and driven by the actual datasets and rule packs expected to be rendered or deployed later by code generation.

## Editing Guardrails

- Preserve existing filenames and scope when they already match the target device or dataset.
- Do not create a second device, logsource, or ingest binding file for the same object unless the split is intentional and justified by the logs.
- Do not hand-write or update `deployments/rule-deployments.yaml`; if deployment enablement must change, route that work through the code path that generates deployment manifests.
- Do not map every observed event field; only add the fields and datasets needed to support the intended content.
- Do not name `dataset_id` from vendor habit alone; use the real log pattern plus the existing rule `logsource` taxonomy so the dataset remains a reliable routing key for applicable content.
- Do not invent `index` or `sourcetype`; use values evidenced by the actual SIEM logs or explicit SIEM metadata supplied with the events.
- Do not add `filters/` or `overrides/` just because they exist in the architecture; create them only when the requested tenant behavior needs them.
- Do not treat `artifacts/` as source input; use it only as rendered output context if needed.
- Keep YAML formatting and key names aligned with nearby tenant files.

## Recheck Checklist

Before finishing, verify all of the following.

- Every new or updated file under `tenants/<tenant>/` matches the repository's current format.
- `device_id` is consistent across `devices/`, `logsources/`, `bindings/ingest/`, and `bindings/fields/`.
- Every `dataset_id` in `bindings/ingest/` exists in the related `logsources/` file.
- Each `dataset_id` in the logsource is justified by both the real log pattern and the relevant existing rules that may apply to it.
- Each `index` and `sourcetype` in `bindings/ingest/` is traceable to the actual SIEM logs or explicit event metadata.
- `tenant.yaml`, `devices`, `logsources`, and `bindings/ingest` match the current schemas under `schema/tenants/`.
- Field binding changes were handled through `$tenant-field-binding-writer`.
- Canonical mapping gaps were handled through `$detection-mapping-ocsf` before finalizing tenant bindings.
- No hand-written deployment manifest was introduced as part of tenant onboarding from logs.

## Logging

Create or update a markdown log under `log/YYYY-MM-DD/tenant-config-from-siem-logs/`.

- Use the current local date for `YYYY-MM-DD`.
- Use a short filename such as `<task-slug>.md`.
- Include the request summary, tenant files changed, inferred device and dataset structure, observed ingest evidence such as `index` and `sourcetype`, delegated field binding or mapping work, validation run or skipped, and assumptions.

## Validation

Run the smallest useful verification after edits.

- Run `python project-root/main.py validate-tenant --tenant-id <tenant>` when the runtime is available.
- If the runtime validator cannot be run, manually re-read the touched tenant files and state that automated validation was not completed.

## Output Expectations

When you perform the task, report briefly:

- which tenant files were created or updated
- which device and dataset structure was inferred from the SIEM logs
- which existing rules or rule `logsource` patterns informed the chosen `dataset_id`
- which observed `index` and `sourcetype` values were used for ingest binding
- whether `$tenant-field-binding-writer` was used
- whether the mapping layer also had to be updated
- whether automated validation was run
