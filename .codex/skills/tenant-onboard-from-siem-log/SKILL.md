# Tenant Onboard From SIEM Log

Use this skill when one SIEM log file or event sample should drive both:

- tenant configuration under `tenants/<tenant>/`
- tenant field bindings under `tenants/<tenant>/bindings/fields/`

Do not use this skill to create or edit `tenants/<tenant>/deployments/rule-deployments.yaml`; deployment manifests are generated by repository code.

This skill is an orchestration skill. It should reuse existing skills instead of duplicating their logic.

## Skills To Use

Use the skills in this order.

1. Use `$tenant-config-from-siem-logs` to determine or update tenant config from the SIEM log input.
2. Use `$tenant-field-binding-writer` to create or update `bindings/fields/*.yml` from the same SIEM log input and the resolved rule pack.
3. Use `$detection-mapping-ocsf` only if the field binding step finds missing or insufficient canonical mappings in `mappings/detections/`.

## Shared Input Rule

Treat the SIEM log file or event sample as a shared source for both skills.

- Read the log input once first and keep a compact working summary.
- Reuse the same sampled events for tenant config and field binding.
- Do not reload the full log separately for each skill unless the first sample was insufficient.
- Keep notes on the observed `device`, `dataset`, `index`, `sourcetype`, and candidate SIEM fields so both steps operate on the same evidence.

## Large Input Handling

Keep context bounded for large log inputs.

- Start with about 200 lines or 16 KB from the input file.
- Prefer the header plus a few representative events.
- Expand sampling only if the current slice is not enough to determine tenant config or field binding decisions.
- State clearly when the log was sampled rather than read in full.

## Orchestration Workflow

Follow this order.

1. Inspect the tenant state under `tenants/<tenant>/` to see whether the device, logsource, ingest binding, or field binding already exists.
2. Sample the SIEM log input and extract the minimum shared evidence:
   `device/product hints`, `dataset split`, `index`, `sourcetype`, repeated field names, and representative event semantics.
3. Use `$tenant-config-from-siem-logs` logic to decide whether `tenant.yaml`, `devices/`, `logsources/`, or `bindings/ingest/` must be created or updated.
4. Identify the relevant rule pack and mapping file from the resulting `device_id`, `dataset_id`, and rule `logsource` expectations.
5. Use `$tenant-field-binding-writer` logic on the same sampled log input to create or update `bindings/fields/*.yml`.
6. If the field binding step cannot trace a required rule field to an existing canonical field, use `$detection-mapping-ocsf` to repair the mapping layer, then return to the field binding step.
7. Re-read the touched tenant files and verify that `device_id`, `dataset_id`, `siem_id`, ingest routing, and field binding scope remain consistent.
8. Leave deployment manifest generation to code; if deployment output matters, report what rule pack the onboarding is intended to support without writing `rule-deployments.yaml`.

## Output Contract

When using this skill, report briefly:

- which sampled log evidence was used
- which tenant config files were created or updated
- whether `bindings/fields` was created or updated
- whether `$detection-mapping-ocsf` was needed
- whether automated validation was run

## Guardrails

- Do not duplicate logic already covered by `$tenant-config-from-siem-logs` or `$tenant-field-binding-writer`; delegate conceptually to those workflows.
- Do not read the full SIEM log input by default when a representative sample is enough.
- Do not create field bindings before the tenant device, logsource, and ingest context are clear enough.
- Do not add tenant config files that are not justified by the observed log input and intended rule pack.
- Do not create or edit `deployments/rule-deployments.yaml`; that file is generated by code after tenant onboarding inputs are in place.
